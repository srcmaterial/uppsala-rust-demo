1//! Alphabet method is very inaccurate by itself to determine a language, but it supports
2//! other methods (trigrams) and allows to improve accuracy on very short texts.
3//!
4//! It's based on the fact, that some languages may use characters that are not present in others.
5//! (e.g. character `ö` is common for German or Finish texts, but not really expected to be
6//! seen in English or Spanish).
7//!
8//! ## Alogirthm
9//!
10//! * For every single character `c` in a lowecased text:
11//!   * If `c` is relevant for the script?
12//!     * count += 1
13//!     * for every language where `c` is a commonly used character
14//!       * raw_lang_score += 1
15//! * Normalize scores for every language:
16//!   * lang_score = raw_lang_score / count
17//!
18//! ## Pros and cons
19//!
20//! This algorithm is very simple and fast, but has also comes with some disadvantages:
21//! * Character frequencies are not respected (e.g. letter `O` has 7.16% occurrence rate in English and only
22//! 2.24% in German, but according to this alphabet model, both English and German would get +1
23//!   score for letter `O`).
24//! * It does not work always well: here are some examples:
25//!   * `Can you tell me where is Schönheitstraße?` - it's cleary an English sentence with a German
26//!   proper name `Schönheitstraße`, but the model gives 34 scores to German and only 30 to
27//!   English, because of the letters `ö` and `ß`, which are present in German, but not in English.
28//!   * `Façade` - is a valid English word. But English gets panished because of the untypical
29//!   character `ç` that was inherited from French.
30//!
31//! The generic (agnostic to a script) implementation and algorithm can be found in the [common] module.
32
